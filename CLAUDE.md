# valid4j

Zero-dependency Java 21 library for typed error accumulation via applicative validation, inspired by Cats' `Validated`.

## Commands

| Check runtime deps | `./gradlew dependencies --configuration runtimeClasspath` |

## Architecture

```
src/main/java/io/github/jpalmerr/valid4j/
  Validated.java       -- Core sealed interface: Valid<E,A> | Invalid<E,A>. Contains combine (arities 2-8) and sequence.
  ValidatedNel.java    -- Convenience layer: factories and combiners for Validated<NonEmptyList<E>, A>. No semigroup param needed.
  NonEmptyList.java    -- Immutable non-empty list (record). Used as the error accumulation type.
  Semigroup.java       -- @FunctionalInterface for associative binary combination. Drives error merging in combine/sequence.
  functions/           -- Function3 through Function8 @FunctionalInterface types (java.util.function only goes to BiFunction).
src/main/java/module-info.java  -- JPMS module: exports valid4j and valid4j.functions packages.

src/example/java/     -- Runnable examples (separate Gradle source set, depends on main).
src/test/java/        -- Unit + property tests.
```

## Key Design Decisions

- **No `flatMap` on `Validated`** -- by design. `combine` is for independent validations (applicative). `andThen` is for sequential dependent steps. This mirrors Cats.
- **Type parameter convention**: `E` is always the error type. Value types use `A, B, C, D, F, G, H, I` -- the letter `E` is skipped in value sequences to avoid collision.
- **`Validated.combine` takes an explicit `Semigroup`**. `ValidatedNel.combine` is the convenience overload that hardcodes `Semigroup.nonEmptyList()`.
- **All public types are null-hostile**: constructors and methods throw `NullPointerException` with descriptive messages on null input.

## Code Style

- **Formatter**: Google Java Format via Spotless. Run `./gradlew spotlessApply` before committing.
- **Compiler strictness**: `-Xlint:all -Werror` on main, test, and example source sets. No warnings allowed.
- **Javadoc**: all public API has Javadoc. Internal comments use `// -----` section dividers.
- **Test naming**: `methodUnderTest_scenario_expectedBehavior` (snake_case segments).
- **Test libraries**: JUnit 5 + AssertJ assertions + jqwik for property-based tests.
- **No mocks**: tests use real instances throughout.

## Testing

- **Unit tests**: `src/test/java/` -- one test class per production class, plus `RealWorldExampleTest` and `ValidatedPatternMatchTest`.
- **Property tests**: `PropertyTest.java` using jqwik. Covers semigroup associativity, functor laws, andThen monad laws, and NEL round-trip.
- **Test structure**: tests grouped by method/operation with `// -----` section dividers matching the production code style.
- Run a single test class: `./gradlew test --tests "io.github.jpalmerr.valid4j.ValidatedTest"`

## Gotchas

- **`Validated.combine` vs `ValidatedNel.combine`**: `Validated.combine` requires an explicit `Semigroup` parameter (second-to-last arg). `ValidatedNel.combine` omits it (hardcodes NEL semigroup). Easy to mix up which overload you're calling.
- **`.jqwik-database` file**: generated by jqwik at project root. Already in `.gitignore`.
- **`example` source set**: compiled with the same `-Xlint:all -Werror` flags as main. Changes to examples can fail the build on warnings.
- **Gradle 8.12** with Java 21 toolchain. The project will not compile on Java < 21 (sealed interfaces, record patterns, pattern matching in switch).
- **Phantom type parameters on `Valid`/`Invalid`**: `Valid<E, A>` has an unused `E` and `Invalid<E, A>` has an unused `A`. Two `Valid` instances with different `E` types but the same value are `equals()` due to type erasure. This is documented in the Javadoc but can surprise.
- **Zero-dependency constraint**: Enforced at build time by `verifyZeroDependencies` task. Adding any runtime dependency will fail the build.
